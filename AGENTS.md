# AGENTS.md — Instructions for autonomous coding agents

This repository is a **reference / RAG corpus**, not an application. Its
purpose is to teach an AI assistant how to write, review, and debug Cisco
Catalyst Center (CC) templates without inventing syntax the platform does
not support.

If you are an autonomous coding agent (Copilot, Claude, GPT-based agent,
etc.) operating on this repo, follow the conventions below.

## 1. First-load order

When a user asks a Catalyst Center template question:

1. Read [INDEX.md](INDEX.md) to discover every available document, its
   engine, kind, and topic.
2. Read [README.md](README.md) for the routing table that maps user
   intents to specific files.
3. Check [prompts/README.md](prompts/README.md) — if the user's request matches
   a curated prompt (authoring / review / debug / pattern), load that prompt
   file and follow its **Agent recipe** and **Output contract** verbatim.
4. Read the **engine-specific rule book** before answering:
   - Velocity → [rules/velocity/constraints.md](rules/velocity/constraints.md)
   - Jinja2   → [rules/jinja2/constraints.md](rules/jinja2/constraints.md)
5. Before proposing code, **check platform limits** in
   [guardrails/jinja2/what-cc-cannot-do.md](guardrails/jinja2/what-cc-cannot-do.md)
   or [guardrails/velocity/what-cc-cannot-do.md](guardrails/velocity/what-cc-cannot-do.md).
6. When a user asks about a Template Editor UI element shown in a screenshot,
   consult [images/README.md](images/README.md) to map topic → image path.

## 2. Engine detection

Determine the engine from explicit user signals first; fall back to
syntax cues only when no signal is available.

| Signal                                       | Engine    |
|----------------------------------------------|-----------|
| File extension `.vm`                         | Velocity  |
| File extension `.j2`                         | Jinja2    |
| User mentions "Velocity" / DNAC Velocity     | Velocity  |
| User mentions "Jinja", "Jinja2", "j2"        | Jinja2    |
| `#set( $x = … )`, `$var`, `#foreach`         | Velocity  |
| `{% set x = … %}`, `{{ var }}`, `{% for %}`  | Jinja2    |
| Ambiguous                                    | **Ask**   |

Composite descriptors are YAML (`*-Composite.yml`) and reference other
templates of either engine.

## 3. Frontmatter contract

Every `.md` file in this repo carries YAML frontmatter:

```yaml
---
title:   <human title>
engine:  jinja2 | velocity | shared | none
kind:    rules | guardrails | docs | navigation | meta
topic:   constraints | variable-types | common-mistakes | …
path:    <repo-relative path of this file>
---
```

Use these fields for filtering when building a retrieval pipeline. Engine
+ kind is enough to narrow most queries to ≤3 files.

## 4. Citation conventions

When citing this corpus, always include the **path and the section
anchor**, not a paraphrase. Examples:

- `rules/jinja2/constraints.md §8.5 — Catalyst Center–specific markers`
- `guardrails/velocity/common-mistakes.md "Bad: # parse" entry`

This lets a reviewer follow the link and verify the claim.

## 5. Catalyst Center platform invariants

Memorise these — they are violated frequently in code generated by
language models trained on generic Velocity or Jinja2:

- `!` is **forwarded as IOS config**, not treated as a template comment.
  Use `{# … #}` (Jinja2) or `## …` / `#* … *#` (Velocity).
- `#MODE_ENABLE` / `#MODE_END_ENABLE` wraps privileged-EXEC commands.
- `<MLTCMD> … </MLTCMD>` bundles a multi-line CLI block (banners, certs,
  key-chains).
- `! @ start-ignore-compliance` / `! @ end-ignore-compliance` excludes
  enclosed CLI from compliance scans.
- PnP onboarding templates **cannot** use bind variables or `__*` system
  variables (the device is not yet in Inventory).
- Composite templates (`*-Composite.yml`) are DayN-only.
- Velocity: `#parse` and `#include` are **disabled**.
- Jinja2: `{% break %}` / `{% continue %}` are unavailable
  (`loopcontrols` extension is off).
- Jinja2: undefined references render as `""` (non-strict mode),
  **unlike** Ansible's default.
- Jinja2: CC accepts `&&`, `||`, `!` in addition to `and`, `or`, `not`.
- Jinja2: string concatenation uses `~`; `+` on strings is arithmetic.

## 6. What you must not do

- **Do not modify files under `examples/`**. They are canonical
  in-production templates and serve as ground truth.
- **Do not invent CC-specific markers** beyond those listed in this
  document and in `rules/*/constraints.md`.
- **Do not propose `{% include %}` / `{% extends %}` / `{% import %}`
  for Velocity** — they are Jinja2-only.
- **Do not assume strict undefined** in Jinja2 examples generated for
  Catalyst Center; show defensive guards (`if x is defined`).
- **Do not output bash/python scripts when asked for a template** —
  authors expect `.j2` or `.vm` content runnable inside the Template
  Editor.

## 7. When in doubt

Ask the user which engine, which template type (Regular / Composite /
PnP / DayN), and which CC version. The answers narrow the search space
sharply.
